web apps

dashboards

APIs

Much safer for deployment.

Very useful for debugging.

│
├── venv/
├── .env
├── requirements.txt
├── manage.py
│
├── project/
│   ├── settings.py
│   ├── urls.py
│   ├── asgi.py
│   └── wsgi.py
│
├── apps/
│   ├── users/
│   ├── payments/
│   ├── dashboard/
│
├── static/
├── media/
└── templates/

This structure helps when projects grow large.

Run:

Add to settings.py:

INSTALLED_APPS = [
    ...
    'django.contrib.sites',

    'allauth',
    'allauth.account',
    'allauth.socialaccount',
]
Add Site ID
SITE_ID = 1
Add Authentication Backends
AUTHENTICATION_BACKENDS = [
    'django.contrib.auth.backends.ModelBackend',
    'allauth.account.auth_backends.AuthenticationBackend',
]
Add URLs

In urls.py:

urlpatterns = [
    path('accounts/', include('allauth.urls')),
]

Below are 15 common mistakes Django developers make and how to avoid them.

1️⃣ Leaving DEBUG = True in Production

In Django, DEBUG=True exposes sensitive error information.

Bad:

DEBUG = True

Correct for production:

DEBUG = False
ALLOWED_HOSTS = ["yourdomain.com"]

Always move this to environment variables.

2️⃣ Hardcoding the Secret Key

Many developers leave this inside settings.py.

Bad:

SECRET_KEY = "django-insecure-123456"

Better:

Use .env

SECRET_KEY=your_real_secret_key

Then load it using django-environ.

3️⃣ Not Using a Virtual Environment

Installing everything globally can break projects.

Always use:

python -m venv venv

Then activate it.

4️⃣ Not Using requirements.txt

Without it, deployments fail.

Always save dependencies:

pip freeze > requirements.txt

Deployment platforms require this.

5️⃣ Using SQLite in Production

SQLite is great for development but not for production.

Instead use PostgreSQL.

Example:

DATABASES = {
    "default": env.db()
}

Most cloud platforms provide PostgreSQL.

6️⃣ Static Files Not Configured Properly

Many Django apps break after deployment because of static files.

Use WhiteNoise.

In settings.py:

STATIC_URL = "/static/"
STATIC_ROOT = os.path.join(BASE_DIR, "staticfiles")

Run:

python manage.py collectstatic
7️⃣ Not Using a Production Server

Running Django like this in production is wrong:

python manage.py runserver

Instead use Gunicorn.

Example:

gunicorn projectname.wsgi
8️⃣ Huge views.py Files

Many beginners put everything in one file.

Bad:

views.py → 2000 lines

Better structure:

views/
    auth_views.py
    dashboard_views.py
    payment_views.py
9️⃣ No Custom User Model

Many developers regret not creating one early.

Best practice:

from django.contrib.auth.models import AbstractUser

class User(AbstractUser):
    phone = models.CharField(max_length=20)

Then in settings:

AUTH_USER_MODEL = "users.User"
🔟 Not Using Django Apps Properly

Large projects should separate features.

Example:

apps/
    users/
    payments/
    memberships/
    dashboard/

This keeps code clean.

1️⃣1️⃣ Not Handling Media Files

Uploads like images break in production.

Add in settings.py:

MEDIA_URL = "/media/"
MEDIA_ROOT = BASE_DIR / "media"
1️⃣2️⃣ Not Using Class-Based Views

Many beginners only use function views.

Better approach:

from django.views.generic import ListView

This reduces code dramatically.

1️⃣3️⃣ Not Securing Forms

Always include CSRF protection.

{% csrf_token %}

Luckily Django enables this by default.

1️⃣4️⃣ Not Optimizing Database Queries

Many developers cause slow websites.

Bad:

users = User.objects.all()

Better:

users = User.objects.select_related("profile")

This avoids N+1 query problems.

1️⃣5️⃣ No Logging System

Production apps should log errors.

Example:

LOGGING = {
    "version": 1,
    "handlers": {
        "file": {
            "class": "logging.FileHandler",
            "filename": "django.log",
        },
    },
}
